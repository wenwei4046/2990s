## What this repo is

Production build of 2990's Home POS + Backend — a Malaysian furniture retailer with a "honest pricing" brand (every Model has its own per-Model pricing, but no upsells / sales / strikethroughs). Two end-user apps + one shared API:

| App | User | Device | Status |
|---|---|---|---|
| **POS** (`apps/pos`) | Sales staff (4 today: AW, JM, RL, SN) | Tablet primary, desktop counter | not built |
| **Backend** (`apps/backend`) | Order Coordinator, future Finance/Owner | Desktop primary | not built |
| **API** (`apps/api`) | Both apps | CF Workers global edge | not built |

Single Supabase Postgres (NEW project, Singapore — **NOT** Venture's `gixpptmfuryskbwkmiwz`).

## Project conventions

### Money
- All prices stored as `INTEGER` representing **whole MYR** (e.g. `2990` = RM 2,990).
- No sen-level pricing in 2990's business model. Don't introduce a `numeric(10,2)` column.

### Order IDs
- Format: `SO-XXXX`, 4-digit number from a Postgres sequence starting at 2050.
- Generated by `next_order_id()` SQL function (see `db/seed-libraries.sql`).
- Used in WhatsApp confirmations, signs, customer-facing surfaces — keep human-readable.

### Brand voice (in any new copy)
- Warm, sincere, calm. No hype. No urgency. No "Limited time!". No emoji.
- Sentence case for everything. Title Case only for tagline-style headlines.
- Body type uses `#221F20` (`--c-ink`), never pure black.
- Bilingual: EN-only for pilot. 中文 toggle (`lang === 'cn'`) wired in prototype but defaults off — don't remove the wiring.

### Icons
- Lucide React only. Stroke 1.75. Sizes: 16 / 20 / 24 / 32 / 40px.
- Never emoji. Never another icon set. Never mix.

### Showroom
- One showroom at MVP: "Showroom KL" (UUID `aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa`).
- `staff.showroom_id` is nullable — coordinators with NULL oversee all showrooms.
- `orders.showroom_id` is NOT NULL — every order is placed at exactly one showroom.
- POS topbar's "POS · Showroom KL" should read from `staff.showroomId` lookup, not be hardcoded — it'll be dynamic when a 2nd showroom opens.

### File scoping
- Files referenced by `apps/pos/*` and `apps/backend/*` should never directly import each other.
- Shared types, Zod schemas, pure pricing math → `packages/shared/`.
- Brand tokens + design primitives → `packages/design-system/`.
- Drizzle schema + migrations → `packages/db/` (or top-level `db/` until Phase 0 moves it).

## Server-side pricing recompute — NON-NEGOTIABLE

`POST /orders` MUST re-derive the total from current pricing tables before persisting:
- Sofa lines: sum compartments per `config.modules`, add recliner upgrades per seat, OR substitute bundle price if `detectBundle` matches AND bundle is cheaper.
- Mattress/bedframe lines: lookup `product_size_variants(product_id, size_id).price`.
- Addons: from `addons` table (lift = `price + perFloorItem * floors * items`).

If the client-submitted total differs from the server-computed total by more than 0.5%, **REJECT the request** with HTTP 400 and the diff. Don't trust the POS bundle. The "honest pricing" brand promise breaks the moment a tampered POS submits `total: 0`.

Lift the pure pricing functions into `packages/shared/src/pricing.ts` so client and server use the same code.

## Project red lines (in addition to global)

1. **Don't modify the prototype** to "fix" something unless explicitly asked. The prototype is canonical spec — changes there ripple to every future port. Approved deviations go through the protocol in `UI_REFERENCE.md`.
2. **Don't redesign sofa configurator UI**. Even if you think a different layout is better. Loo finalised this through multiple design reviews. The 22 plan-view PNGs and the snap math are not negotiable.
3. **Don't substitute the stack** in §Stack above. Tailwind, shadcn, react-dnd, Next.js — all rejected with reasons. Use the prototype's CSS classes + tokens.
4. **Don't skip server-side pricing recompute** on `POST /orders`. The whole brand promise depends on it.
5. **Don't seed pricing data** that conflicts with what's in `prototype/pos-data.jsx`. The 4 sofa Models (Noor / Tanah / Rumah / Petang) and 4 mattress / 3 bedframe SKUs already have hand-set per-Model pricing — port those, don't re-imagine them.
6. **Don't expose the Backend portal to non-staff**. RLS is restrictive by default; verify before deploy.
