# Money-Side Deep Audit — 2026-06-11

Scope: `apps/api/src/routes/sales-invoices.ts`, `purchase-invoices.ts`, `purchase-returns.ts`,
`delivery-returns.ts`, `apps/api/src/lib/customer-credits.ts`, `lib/post-si-revenue.ts`,
`routes/accounting.ts` (PI post/reverse/resync), SO deposit-credit path in
`routes/mfg-sales-orders.ts`. All amounts in sen/centi (INTEGER columns — `qty INTEGER`,
so `qty * unit_price` arithmetic is integer-safe; no roundSen-class fractional-centi risk found).

Excluded as known (per BUG-HISTORY newest entry, 2026-06-11): SI reopen re-validation (fixed),
SI/DO read-then-write race backstops (no partial unique indexes), PR bare-create clamp-not-reject,
GRN header lock. Doc-number `count(*)`-based races (SI/PI/PRT/DR `nextNum`) are the same known
class and only noted in passing (a duplicated `invoice_number` would additionally make two SIs
share one revenue JE, since all JE idempotency keys on `source_doc_no`).

Every finding below was re-verified at file:line before inclusion. No app code was modified.

## HIGH

### H1 — SO un-cancel never claws back the SO_CANCEL_REFUND deposit credit (double-counted deposit)
- **Where:** `apps/api/src/routes/mfg-sales-orders.ts:2859-2937` (status PATCH), `apps/api/src/lib/customer-credits.ts:280-317` (`creditFromCancelledSo`).
- **What:** Cancelling an SO with deposits writes a positive `SO_CANCEL_REFUND` credit equal to
  Σ `mfg_sales_order_payments` (line 2921-2934). But the status route gates **only the
  CANCELLED transition** (line 2872: `if (body.status === 'CANCELLED' …)`); any transition
  *out of* CANCELLED (CANCELLED→CONFIRMED etc.) rides through with no guard and **no contra
  credit entry** — there is no `SO_REOPEN_CONTRA` analogue of the SI-side
  `reverseCancelledSiCredit`. After un-cancel: the SO is live again with its deposit payments
  intact AND the customer keeps the full deposit as spendable credit (auto-applied to their next
  SI by `applyCustomerCreditToSi`). Deposit is double-counted.
- **Note:** `creditFromCancelledSo`'s idempotency (`already_credited` on any existing
  `SO_CANCEL_REFUND` row, lines 287-295) also means a legitimate *second* cancel after
  reopen-and-repay would NOT credit again — the inverse defect of the SI-side netting design.
- **Fix direction:** mirror the SI pattern — block reopen, or add `SO_REOPEN_CONTRA` written on
  any CANCELLED→active transition, and net it in `creditFromCancelledSo` like
  `creditFromCancelledSi` does (customer-credits.ts:216-225).

### H2 — Cancelling an over-paid SI double-credits the overpay excess
- **Where:** `apps/api/src/lib/customer-credits.ts:204-238` (`creditFromCancelledSi`),
  triggered from `apps/api/src/routes/sales-invoices.ts:1060-1077`.
- **What:** Pay 150 on a 100-total SI → `reconcileSiOverpay` books `OVERPAY +50`
  (customer-credits.ts:155-197). Cancel the SI → `creditFromCancelledSi` credits the **full**
  `paid_centi` (+150). Its standing check (lines 216-225) nets only
  `SI_CANCEL_REFUND`/`SI_REOPEN_CONTRA` — it ignores the live `OVERPAY` entry, and
  `reconcileSiOverpay` skips CANCELLED invoices (line 167) so it never corrects either.
  Customer credit = 200 for 150 cash received. (Reopen happens to land correct again because the
  contra removes only the 150; the over-credit exists exactly while the SI stays cancelled —
  i.e. in the state where the credit is spendable.)
- **Fix direction:** in `creditFromCancelledSi`, credit `paidCenti − Σ live OVERPAY entries for
  this SI` (or fold OVERPAY rows into the standing-netting query).

### H3 — SI payment add/delete has no CANCELLED guard → cash ledger drifts under a frozen cancel-credit
- **Where:** `apps/api/src/routes/sales-invoices.ts:870-911` (POST `/:id/payments` — loads
  `select('id')` only, no status check) and `:913-926` (DELETE — same).
- **What:** The legacy PATCH `/:id/payment` blocks CANCELLED (line 1122), but the primary
  ledger endpoints don't. On a CANCELLED SI:
  - **Delete a payment** → `recomputePaid` lowers `paid_centi`, `reconcileSiOverpay` no-ops
    (skips cancelled) — but the `SI_CANCEL_REFUND` credit written at cancel time was sized to
    the *old* `paid_centi`. Cash record gone, customer keeps the full credit. Credit no longer
    backed by money.
  - **Add a payment** → `paid_centi` rises on a cancelled invoice; no credit, no overpay entry;
    if the SI is later reopened, the reopen contra reverses only the historical credit and
    nothing re-derives status from the ledger (see M1).
- **Fix direction:** 409 both payment routes when status = CANCELLED (same message as the
  legacy route), or make the cancel-credit re-reconcile on any payment mutation.

## MEDIUM

### M1 — SI fully covered by auto-applied credit keeps status SENT (status not ledger-derived on the credit path)
- **Where:** `apps/api/src/routes/sales-invoices.ts:336-358` (POST `/` Edge #11 block) and
  `:562-583` (`/from-dos`); `apps/api/src/lib/customer-credits.ts:110-137`.
- **What:** `applyCustomerCreditToSi` inserts the `method='credit'` payment row and **manually
  increments `paid_centi`** (lines 132-135) but never touches `status`/`paid_at`, and neither
  create path calls `recomputePaid` afterwards. An invoice 100% covered by existing credit is
  created PAID-in-substance but shows `SENT` (and `paid_at` null) until some later payment event
  happens to run `recomputePaid`. Same gap on reopen: CANCELLED→SENT (line 1080-1106) restores
  the invoice without re-deriving PARTIALLY_PAID/PAID from the surviving payment rows.
- **Fix direction:** call `recomputePaid(sb, siId)` after `applyCustomerCreditToSi` in both
  create paths and after the reopen transition.

### M2 — Legacy quick-payment endpoint skips overpay reconciliation
- **Where:** `apps/api/src/routes/sales-invoices.ts:1113-1136` (PATCH `/:id/payment`).
- **What:** Inserts the payment + `recomputePaid`, but unlike POST `/:id/payments` (line 908) it
  never calls `reconcileSiOverpay`. An overpayment recorded from the Outstanding page books **no
  OVERPAY credit** — the excess silently vanishes from the customer's balance until some
  unrelated payment add/delete on the same SI happens to reconcile it.
- **Fix direction:** add the same `reconcileSiOverpay` best-effort call.

### M3 — PI discount handling differs across all four line-creation paths (totals drift)
- **Where:** `apps/api/src/routes/purchase-invoices.ts`
  - POST `/` (manual create, lines 341-356): `total = qty * unit` — client `discountCenti` is
    **silently dropped** (no `discount_centi` column written either) → manual PI totals
    overstated whenever the form sends a discount.
  - POST `/from-grn-items` (lines 624-647): stores `discount_centi` (line 646) but
    `line_total_centi = qty * unit` (line 632) **excludes it**; header subtotal likewise
    (line 601).
  - POST `/from-grn` (lines 722, 750): `_remaining * unit − discount_centi` subtracts the
    **full GRN-line discount** even when billing a partial remaining — and a second
    `/from-grn` pass over the same line subtracts the same full discount **again**.
  - PATCH `/:id/items/:itemId` (line 948): recomputes `lineTotal = qty*unit − discount` — so
    editing *any* field of a `/from-grn-items` line (even notes is safe, but qty/price/discount
    passthrough isn't) silently shifts its total by the stored, previously-unapplied discount,
    then `recomputePiTotals` moves the header total with no operator-visible cause.
- **Effect:** the same GRN line billed via different paths yields different AP totals; GL posting
  (`postPiAccounting` keys on `total_centi`) and `paid_centi`-derived status inherit the drift.
- **Fix direction:** pick ONE rule (pro-rated discount per billed qty is the only one that is
  multi-PI-safe), apply in all four paths.

### M4 — PI payment is a read-modify-write with no overpay clamp and no ledger
- **Where:** `apps/api/src/routes/purchase-invoices.ts:422-444` (PATCH `/:id/payment`).
- **What:** `newPaid = c0.paid_centi + amount` read-then-write — two concurrent payments lose
  one update (SI side uses a payments table + recount; PI has no payments table). No clamp:
  `paid > total` is accepted, instantly flips status PAID, excess is untracked (no AP-credit
  concept). Also `newPaid >= total` with `total = 0` marks an empty PI PAID on any payment.
- **Fix direction:** at minimum reject `amount > total - paid` (reject-don't-normalize), or move
  PI payments to a ledger table like the SI side.

### M5 — PI header PATCH bypasses the edit-lock and never resyncs the GL
- **Where:** `apps/api/src/routes/purchase-invoices.ts:803-821`.
- **What:** PATCH `/:id` has no `piLocked` call — supplier, invoice/due dates, currency are
  editable on a CANCELLED or PAID PI (lines routes all lock; header doesn't). Changing
  `supplier_id` after `postPiAccounting` leaves the AP JE's party (supplier code/name snapshot)
  pointing at the old supplier with no resync. `tax_centi` additionally has no write path at all
  yet is added into `total_centi` by `recomputePiTotals` (line 43) — dead but live-wired.
- **Fix direction:** apply `piLocked` to the header PATCH; resync accounting if supplier/date
  change while a live PI JE exists.

### M6 — PI add-line never resyncs posted accounting (edit/delete do)
- **Where:** `apps/api/src/routes/purchase-invoices.ts:824-925` (POST `/:id/items`).
- **What:** PATCH item (line 1004-1007) and DELETE item (line 1040-1043) both call
  `resyncPiAccounting` after changing the total; the add-line route ends at `recostForPi`
  (line 923) with no resync. A PI that was posted to the GL under-states AP by every added line
  until some later edit/delete happens to resync.
- **Fix direction:** add the same best-effort `resyncPiAccounting` call.

### M7 — SI append-from-DO has no same-customer guard
- **Where:** `apps/api/src/routes/sales-invoices.ts:593-638` (POST `/:id/items/from-do/:doId`).
- **What:** `/from-dos` enforces one customer (lines 413-428); the detail-page append path
  checks only that SI ≠ CANCELLED and DO ≠ CANCELLED. Customer B's delivered lines can be
  appended into customer A's invoice: B's Pending pool is consumed, A's AR/revenue is raised,
  and B can never be billed for those units (remaining hits 0). Also skips credit auto-apply /
  `recomputePaid` (consistent with other line routes, but the cross-customer hole is the issue).
- **Fix direction:** compare DO `debtor_code/debtor_name` against the SI header; 400
  `mixed_customers` like `/from-dos`.

### M8 — DR line CRUD has no terminal-status lock (parity gap with consignment-returns `returnLineLock`)
- **Where:** `apps/api/src/routes/delivery-returns.ts:1079-1126` (add), `:1128-1200` (edit),
  `:1202-1211` (delete).
- **What:** BUG-2026-06-07-002 #3 added `returnLineLock` (REFUNDED / CREDIT_NOTED / CANCELLED →
  409) to consignment-returns; delivery-returns line routes have **no status check at all**.
  On a REFUNDED or CREDIT_NOTED DR, line add/edit/delete rewrites `refund_centi`
  (`recomputeTotals` line 97) **after settlement** and re-books physical stock via
  `resyncInventoryForReturn`. On a CANCELLED DR the stock resync self-protects (target = 0,
  lines 357-380) but `recomputeTotals` still rewrites the money rollups of a dead document.
  (The 2026-06-10 `dr_cancelled_final` fix covers only the status route, line 1245.)
- **Fix direction:** port `returnLineLock` (REFUNDED / CREDIT_NOTED / CANCELLED) into the three
  DR line routes.

### M9 — DR convert-from-DO drops the DO line discount → refund overstated
- **Where:** `apps/api/src/routes/delivery-returns.ts:994` (`discountCenti: 0` in the
  `/from-do(s)` line build) vs `apps/api/src/routes/sales-invoices.ts:536` (SI `/from-dos`
  carries `line.discountCenti`).
- **What:** `buildItemRow` sets `line_total = qty*unit − discount` and `refund_centi`
  defaults to `line_total` (lines 585-589). The convert path hardcodes discount 0, so a
  discounted DO line returned via the picker records `refund_centi` at the **full undiscounted
  price** — more than the customer ever owed for it. The bulk POST `/` path uses whatever the
  client sends, so the two DR creation paths disagree for the same DO line.
- **Fix direction:** pass the DO line's (qty-pro-rated) discount like the SI convert does.

### M10 — Item DELETE routes are not scoped to their parent document
- **Where:** `sales-invoices.ts:803` (`delete().eq('id', itemId)` — no
  `eq('sales_invoice_id', id)`), `purchase-invoices.ts:1022`, `purchase-returns.ts:1063`,
  `delivery-returns.ts:1204`. (Contrast: SI payment DELETE *does* verify the doc match,
  sales-invoices.ts:915-917.)
- **What:** DELETE `/X/:idA/items/:itemIdOfB` removes B's line while every follow-up
  (recomputeTotals / GL resync / GRN release / stock resync) runs against A. B is left with a
  stale total, an unsynced revenue/AP JE, and (PR/DR) stock that was never compensated. Needs a
  malformed/mismatched client call, but a single UI bug or stale tab turns it into silent money
  + stock corruption.
- **Fix direction:** add the parent-id equality filter (and 404 when no row deleted) on all four,
  + the same scoping on the item PATCH routes (they read by itemId alone too).

## LOW / OBSERVATIONS

### L1 — Customer credit ledger is currency-blind
`customer_credits` (migration 0110) has no currency column; SI `currency` is a free header field
(`sales-invoices.ts:298,653`). Credits earned on a MYR invoice apply 1:1 to a USD invoice
(`applyCustomerCreditToSi` compares raw centi). Harmless while everything is MYR; a latent trap.

### L2 — SI header PATCH allows debtor/currency rewrite anytime, including CANCELLED
`sales-invoices.ts:641-680` — no status gate; `debtor_code` is rewritable after credit was
auto-applied (the APPLIED_TO_SI row stays keyed to the old debtor) and the eventual
cancel-credit goes to the NEW debtor: same SI can debit one customer's credit pool and refund
another's. Also editable on a cancelled invoice.

### L3 — `recomputePaid` status edges
`sales-invoices.ts:856-866`: a zero-total SI with payments sits at PARTIALLY_PAID forever
(`total_centi > 0` required for PAID); any recompute collapses OVERDUE back to SENT when unpaid.

### L4 — DR "REFUNDED" moves no money
`delivery-returns.ts:1227-1306` — REFUNDED stamps `refunded_at` only. No customer-credit entry,
no payment/AR adjustment, no cap against what the customer actually paid; `refund_centi` is
informational. Because the shared Pending pool means only *uninvoiced* delivered units are
returnable, no revenue reversal is needed — but a cash refund recorded here leaves zero trace in
any money ledger. Design gap to confirm with the owner rather than a code defect.

### L5 — PR refund never reaches AP/GL; PR header PATCH has no terminal lock
`purchase-returns.ts` — `refund_centi` + `credit_note_ref` are tracking fields only (no JE, no
PI/AP adjustment; if a GRN line was already fully invoiced, returning it produces no AP
claw-back — supplier credit note is manual). Header PATCH (lines 810-826) permits editing
supplier/date/credit_note_ref on COMPLETED/CANCELLED PRs while the line routes are locked
(`prLineLock`, line 833).

### L6 — `purchase-invoices.ts` PATCH `/:id/post` 500s instead of clean 409 on a CANCELLED PI
Line 412-416: the `.neq('status','CANCELLED')…single()` returns a PGRST116 error when the row is
cancelled → `post_failed` 500, the `cannot_post` 409 branch is unreachable. Cosmetic.

## Cross-checks that came back CLEAN
- `postSiRevenue` / `reverseSiRevenue` / `resyncSiRevenue` idempotency chain (active-JE keyed,
  reversal keyed on `reversed_by_je`, CANCELLED short-circuit in resync) — no double-post path
  found through create / from-dos / add-line / edit / delete / cancel / reopen.
- SI cancel race (conditional `neq('status','CANCELLED')` update) and PI/PR cancel equivalents —
  single-winner semantics hold.
- Credit lifecycle on a *normally* paid-by-credit SI: apply (−) → cancel (+paid) → reopen (−)
  nets correctly; `applyCustomerCreditToSi` idempotency via the `method='credit'` payment row is
  safe because the operator payment schema enum cannot create `method='credit'` rows
  (sales-invoices.ts:833-845).
- `recomputeGrnInvoiced` / `verifyGrnLinesNotOverInvoiced` / `adjustGrnReturnedQty` recount-from-
  live models converge; clamps verified.
- Integer-money arithmetic: `qty` is INTEGER (migration 0042), all `_centi` columns INTEGER —
  no fractional-sen storage paths.
