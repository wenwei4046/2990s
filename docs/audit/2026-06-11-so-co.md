## CRITICAL

### C-1. SO un-cancel mints free money вЂ” cancel-credit is never clawed back and the cancel transition is reversible

- **Where:** `apps/api/src/routes/mfg-sales-orders.ts:2859-2937` (PATCH `/:docNo/status`), `apps/api/src/lib/customer-credits.ts:280-317` (`creditFromCancelledSo`).
- **Code:** the status PATCH accepts any `mfg_so_status` enum value with no transition matrix. Only the *into*-CANCELLED transition is guarded (`soHasDownstream`, line 2872). `CANCELLED в†’ CONFIRMED` (or any other status) passes straight through. On cancel, `creditFromCancelledSo` mints a `SO_CANCEL_REFUND` customer-credit row equal to ОЈ `mfg_sales_order_payments`. There is **no `SO_REOPEN_CONTRA`** вЂ” the SI side has exactly this pair (`creditFromCancelledSi` + `reverseCancelledSiCredit`, customer-credits.ts:204-272), the SO side only has the mint half.
- **Scenario:** SO-2606-050, deposit RM1,000 in the payments ledger. Operator cancels в†’ customer gets RM1,000 credit (spendable on any future SI via `applyCustomerCreditToSi`). Operator un-cancels (status back to CONFIRMED вЂ” nothing blocks it). The SO is live again, its RM1,000 ledger payment still counts toward its balance, **and** the RM1,000 credit still stands в†’ the same money exists twice. A later re-cancel does NOT re-mint (idempotency on `SO_CANCEL_REFUND` source pair works) but never needs to вЂ” the damage is the standing credit + live order.
- Related hole: `DELETE /:docNo/payments/:id` (line 4406) has no status lock, so payments on a CANCELLED SO can be deleted after the credit was minted вЂ” credit stays at the old sum.
- **Fix direction:** either make SO CANCELLED final (mirror `do_cancelled_final` from BUG-2026-06-11-001 #2), or port `reverseCancelledSiCredit` as `reverseCancelledSoCredit` and fire it on any CANCELLED в†’ non-CANCELLED transition.
- **Severity:** CRITICAL В· **Confidence:** high (all paths read; no claw-back code exists anywhere вЂ” grepped `SO_CANCEL_REFUND` repo-wide).

### C-2. POS anti-tamper drift gate does not cover `discountCenti` вЂ” a tampered tablet can zero out any order

- **Where:** `mfg-sales-orders.ts:1868` (POST `/` line build: `const discount = Number(it.discountCenti ?? 0);`), `:3473` (add-line), `:3654` (line PATCH). Drift rejection (`pricing_drift`, lines 1817-1833 / 3529 / 3721) compares **unit price only**.
- **Code:** `lineTotal = (qty * unit) - discount` with `discount` taken verbatim from the client вЂ” no clamp (can be any size, even negative), no relation to qtyГ—unit, no role gate. The POS-tablet trust boundary (CLAUDE.md non-negotiable: "a tampered POS must never submit a doctored low total") rejects a doctored *unit price* >0.5% off, then happily subtracts an arbitrary client-authored discount.
- **Scenario:** tampered POS submits the correct catalog `unitPriceCenti` (passes drift) plus `discountCenti: 299_900` on a RM3,000 line в†’ SO total RM1.00, header/category totals, margin, balance all follow. Conversely a **negative** discount silently inflates a Backend-entered total.
- Note: the sofa split passes the full discount to the first module line (`:2008`), so the hole rides through the split unchanged.
- **Severity:** CRITICAL (defeats the anti-tamper gate; direct revenue leakage path) В· **Confidence:** high.

## IMPORTANT

### I-1. Master sofa-combo COST spread is dead code since the parens vocabulary unification вЂ” sofa set costs/margins never get the combo price

- **Where:** `mfg-sales-orders.ts:3333` and `consignment-orders.ts:1129` вЂ” `if (!sizeCode.includes('-')) continue; // non-modular`.
- **Code:** the gate was written for the legacy dash module form (`ANNSA-1A-LHF` в†’ sizeCode `1A-LHF`, contains `-`). Since 2026-06-04 the ONE compartment vocabulary is the parens form (`packages/shared/src/sofa-build.ts:298-321` вЂ” "the canonical form IS what renders everywhere; (1A-LHF) survives only inside normalizeCompartmentCode for tolerance"), and the P3 splitter itself emits `item_code = ${baseModel}-${normalizeCompartmentCode(moduleId)}` (`so-sofa-split.ts:111,123`) e.g. `ANNSA-1A(LHF)` в†’ `splitSofaCode` sizeCode = `1A(LHF)` в†’ **no dash в†’ every canonical module line is skipped as "non-modular"**. The whole combo-match + `spreadComboTotal` block below it (3338-3368 / 1134-1164) can no longer fire for any P3-split sofa. Header cost = ОЈ Г -la-carte module costs; the cheaper master combo cost is never applied в†’ `total_cost_centi` overstated, `total_margin_centi`/`margin_pct_basis` understated on every combo-matched sofa SO/CO created since the unification.
- Same stale gate exists on the PO cost path (`mfg-purchase-orders.ts:1038` вЂ” out of audit scope, flagged for the parity sweep).
- **Fix direction:** replace the dash test with "is this sizeCode a known compartment" (e.g. `findModule(normalizeCompartmentCode(sizeCode))` or simply parens-or-known-module), not a dash sniff. Note when reviving: the grouping also merges *separate builds* of the same model into one member pool (no `buildKey` separation) and counts a qty-2 line as ONE module in the match multiset вЂ” both worth fixing in the same pass (see M-9).
- **Severity:** IMPORTANT (cost/margin reporting wrong; selling price unaffected) В· **Confidence:** high from code; not verified against live prod SKU rows.

### I-2. Consignment Return un-cancel skips the inventory resync (the exact DR bug fixed in BUG-2026-06-11-001 #3, not ported to the clone)

- **Where:** `consignment-returns.ts:667-710` (PATCH `/:id/status`).
- **Code:** `resyncReturnInventory` is called **only** `if (body.status === 'CANCELLED')` (line 705). A CANCELLED CRN can be PATCHed back to `RECEIVED`/`INSPECTED` (no `cr_cancelled_final` guard, no resync on the way out). The cancel's balancing OUT stays in the ledger while the doc again says "received back" в†’ stock understated by the whole return until some unrelated line edit happens to re-trigger the self-healing resync. The sales-side DR got "CANCELLED is final (409 `dr_cancelled_final`)" on 2026-06-11; this faithful clone did not.
- (Contrast: the CN status route is safe вЂ” it resyncs on `SHIPPED_STATES.includes(status) || CANCELLED` (`consignment-notes.ts:808`), so a CN un-cancel re-emits its OUT.)
- **Severity:** IMPORTANT В· **Confidence:** high.

### I-3. CO line-price override route has NO admin role gate and NO price-override audit row (SOв†”CO parity drift)

- **Severity:** IMPORTANT (price-control bypass on the consignment book) В· **Confidence:** high.

### I-4. SO/CO price-override route bypasses the downstream lock вЂ” line money mutable after a DO/SI snapshot

### I-5. CO payments POST has no overpayment guard and no slip requirement (SOв†”CO parity drift)

### I-6. Cancelling an SO leaves its earned PWP vouchers redeemable

- **Where:** `mfg-sales-orders.ts:2764-2768` (carried-forward RESERVED в†’ AVAILABLE at create, bound to the customer) and `:2859-2937` (status PATCH вЂ” no `pwp_codes` writes on CANCELLED; grepped: only `mfg-sales-orders.ts` and `pwp-codes.ts` touch the table, neither voids on cancel).
- **Scenario:** customer buys a trigger mattress, SO prints a PWP voucher (AVAILABLE, `source_doc_no` = this SO). Order is cancelled the next day (deposit в†’ credit). The voucher survives вЂ” the customer (or sales) redeems the discounted/free reward on a later order even though the qualifying purchase never happened. Combined with C-1's standing credit, a cancelled order can fund both a credit and a reward discount.
- **Fix direction:** on CANCELLED, delete/void `pwp_codes` where `source_doc_no = docNo AND status = 'AVAILABLE'` (USED ones already burned on another order need a policy decision).
- **Severity:** IMPORTANT В· **Confidence:** high.

### I-7. so-stock-allocation reads are unbounded в†’ silent 1000-row PostgREST truncation degrades allocation at scale

- **Where:** `apps/api/src/lib/so-stock-allocation.ts:98-102` (all lines of all active SOs, no limit), `:118` (`mfg_products` **full-catalog** `select('code, category')`, no filter/limit), `:277-280` (`inventory_balances` by product set, no limit); also `sofa-set-coverage.ts:62-66` (all open batched lots).

- **Severity:** IMPORTANT (latent вЂ” depends on table sizes / configured max-rows; today probably under the cap) В· **Confidence:** medium.

- **PWP voucher claim machinery**: duplicate-code-per-order gate, atomic conditional UPDATE claim, prefetch-failure honesty, orphaned-USED self-heal with exact-match re-claim, rollback on every reject path (drift, slip, cross-category, insert failure), promo one-way backstop, trigger_item_code re-stamp. (The only gap found is cancel-time voiding вЂ” I-6.)
- **`recomputeFromSnapshot`**: NaN-safe `extraAddonAmountRM`, client-0 = "not provided" semantics, sofa fall-back to operator price when modules unpriced (no false rejects), PWP base/combo override layering, cost path isolated from selling.
- **Delivery-fee unit handling**: `delivery_fee_config` + special-model fees are whole-MYR and consistently Г—100 before the sen-domain `computeSoDeliveryFee` (`mfg-sales-orders.ts:2113-2139`); `ОЈ SVC-DELIVERY* lines === deliveryFeeCenti`; `recomputeTotals` folds the header fee back ONLY when no fee lines exist (no double count).
