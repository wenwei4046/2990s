# Frontend Deep Bug Audit — apps/backend/src (2026-06-11)

Scope: React + TanStack Query backend-office app (`apps/backend/src/**`). Read-only audit — no app code modified.
Cross-checked against `docs/BUG-HISTORY.md` (knowns excluded). Backend side-effects verified in `apps/api/src/routes/*` where a finding depends on them.

Context that makes the cache findings real: the QueryClient runs `staleTime: 60_000` and `refetchOnWindowFocus: false`
(`apps/backend/src/main.tsx:13-20`). A missed `invalidateQueries` therefore shows stale data for up to 60 s even across
page navigations — it is not masked by default refetching.

Note: there are no Employees/payroll screens in this app (that module lives in a different repo). POS is a separate app
(`apps/pos`) and was out of scope.

## P1 — money input / money corruption / lost work

### F1. Reformat-on-every-keystroke money inputs — "typing 500 registers 5", field cannot be cleared
The exact bug class `components/MoneyInput.tsx` (header comment, lines 1-23) was built to kill — a controlled
`<input type="number" value={(centi/100).toFixed(2)}>` that re-formats on every keystroke — is still live in at least
seven places. Typing "500" into an empty/0 field: first keystroke commits 500 sen, re-render forces the text back to
"5.00", every further digit is swallowed (`"5.000"` → `Math.round(5*100)` → re-render "5.00"). Clearing is also
impossible (`'' → Number('') = 0 → "0.00"`). Only select-all-and-retype of a value with the same digit count works.

| Site | Line | What it edits |
|---|---|---|
| `components/PaymentsTable.tsx` | 740-748 | Payment amount on SO/DO/SI payment draft rows (real money). Partially masked because `addDraft` seeds the outstanding balance (line 342-351), but any manual edit hits it. |
| `pages/SupplierDetail.tsx` | 2258-2263 | Binding Unit Price |
| `pages/SupplierDetail.tsx` | 3025-3033 | Bulk-bind table unit price |
| `pages/SupplierDetail.tsx` | 3355-3362 | Material binding unit price |
| `pages/ProductModels.tsx` | 1654-1661 | Model bulk-create supplier unit price |
| `pages/Products.tsx` | 2830-2840 | Maintenance priced-option price (RM) |
| `pages/Products.tsx` | 2864-2881 | Maintenance priced-option COST (RM) |

Fix direction: replace each with the existing `MoneyInput` (`bare` mode exists for form fields) — that component was
created precisely for this and already handles blur-commit/Esc/blank.

### F2. SKU CSV import: `parseFloat` silently truncates at a thousands separator → price corruption
`pages/Products.tsx:3848` — `senOf()` uses `parseFloat(rm)` for ringgit columns. A cell that round-tripped through
Excel with thousands formatting (`"1,535.00"`, properly quoted, parsed intact by the RFC4180 parser at 3746+) becomes
`parseFloat("1,535.00") = 1` → **RM 1.00** written to `base_price_sen`/`price1_sen`/sofa seat prices with no warning.
(`Number()` on the legacy `_sen` column at 3849 would at least yield NaN→skip.) This violates the house rule
"reject bad inputs, don't normalize". Fix: validate `/^\d+(\.\d{1,2})?$/` (or strip-and-reject on comma) and surface the
row in `tierErrors`-style failures.

### F3. Mid-edit draft wipe — photo upload / line delete reverts every other line's unsaved edits
- `pages/SalesOrderDetail.tsx:588-600` — the edit-mode seed effect re-seeds **all** `editingDrafts` from the server
  snapshot whenever `items` changes identity.
- Photos on a saved line can only be uploaded/deleted while `isEditing` (`components/SoLineCard.tsx:227`,
  `canMutatePhotos = isSaved && isEditing`; upload at 965, delete at 902-905). `useUploadSoItemPhoto` /
  `useDeleteSoItemPhoto` invalidate `['mfg-sales-order-detail', docNo]` → detail refetch → new `items` → the seed
  effect **overwrites every line's in-progress qty/price/variant edits** with server values. Silent — no warning.
- Same with per-row delete: `rowCallbacks.onRemove` (SalesOrderDetail.tsx:616-623) deletes one line mid-edit,
  invalidation refetches items, and the other lines' unsaved edits revert.
- Same pattern on `pages/DeliveryOrderDetail.tsx:287-301`, which additionally re-seeds **payment drafts** whenever
  `paymentsQ.data` refetches mid-edit.

Fix direction: merge instead of replace on re-seed (preserve drafts whose line still exists and is dirty), or only
seed lines that don't already have a draft.

## P1 — TanStack Query invalidation gaps (stale stock / stale documents after real inventory writes)

All verified against the API routes; staleness window ≤60 s incl. cross-page navigation (see context above).
All in `apps/backend/src/lib/flow-queries.ts` unless noted.

### F4. `usePostGrn` (flow-queries.ts:88-95) → invalidates only `['grns']`
Backend `POST /grns/:id/post` runs `postGrnAndRollup`: writes inventory IN, rolls `received_qty` to the PO, flips PO
status, re-walks SO allocation (`apps/api/src/routes/grns.ts:43-110`). Missing invalidations: `['inventory']`,
`['grn-detail', id]`, `['mfg-purchase-orders']`, `['grns','outstanding-po-items']`, `['mfg-sales-orders']`.
Compare `useCancelGrn` (flow-queries.ts:160-170) which *does* invalidate `['inventory']`. Used by `pages/GrnNew.tsx:520`
(create→post): after saving a receipt, the Inventory page and the PO's received column show pre-receipt numbers.

### F5. GRN item add/update/delete (flow-queries.ts:103-155) → only `['grn-detail', id]` + `['grns']`
On a posted GRN the backend writes the inventory IN per added line and an OUT reversal per deleted line
(`grns.ts:673, 1133, 1462`) and recounts PO `received_qty`. Missing: `['inventory']`, `['mfg-purchase-orders']`,
`['grns','outstanding-po-items']`.

### F6. DO item add/update/delete (`useAddMfgDeliveryOrderItem` / `useUpdateMfgDeliveryOrderItem` / `useDeleteMfgDeliveryOrderItem`) → only DO detail + DO list
On a SHIPPED DO the backend resyncs inventory to the new line shape (extra OUT / give-back IN,
`apps/api/src/routes/delivery-orders-mfg.ts:409-577`) and syncs SO delivered qty (`syncSoDeliveredFromDo`).
Missing: `['inventory']`, `['mfg-sales-orders']` (+ detail), `['mfg-delivery-orders','deliverable-so-lines']`,
`['sales-invoices','invoiceable-do-lines']`, `['delivery-returns','returnable-do-lines']`.
Compare `useUpdateMfgDeliveryOrderStatus` which correctly invalidates `['inventory']`.
Operator-visible: delete a DO line, open the SO → still shows fully delivered / not reconvertible for up to 60 s.

### F7. Purchase Return create / line edits → inventory + GRN rollup stale
PRs are created POSTED with inventory OUT written inline (`apps/api/src/routes/purchase-returns.ts:399-458`), and
line update/delete writes IN/OUT deltas + `grn_items.returned_qty` recount (purchase-returns.ts:326-395).
But `useCreatePurchaseReturn` invalidates only `['purchase-returns']`; `useUpdatePurchaseReturnItem` /
`useDeletePurchaseReturnItem` only PR detail + list. Missing: `['inventory']`, `['grns']`,
`['purchase-invoices','outstanding-grn-items']`. (The from-GRN convert hooks DO invalidate `['grns']` — the manual
create path drifted.)

### F8. `usePostPurchaseConsignmentReceive` (lib/purchase-consignment-receive-queries.ts) → only `['pc-receive']`
Posting books the consignment inventory IN + rolls up to the PC order
(`apps/api/src/routes/purchase-consignment-receives.ts:718`). Missing: `['inventory']`, `['pc-receive-detail', id]`,
`['pc-order']`.

## P2 — payment/aging staleness, dates, double-document, grid layout

### F9. Payment mutations don't refresh aging / list balances
- `useRecordSiPayment` (flow-queries.ts) → `['sales-invoices']` + detail only. `v_ar_aging` derives from
  `sales_invoices.paid_centi` (`packages/db/migrations/0052_accounting_journal_entries.sql:178-198`) → Accounting
  page AR aging stale. Same for `useAddSalesInvoicePayment` / `useDeleteSalesInvoicePayment` (no `['ar-aging']`).
- `useRecordPiPayment` → no `['ap-aging']` (PI payment PATCH updates `paid_centi`,
  `apps/api/src/routes/purchase-invoices.ts:420-442`).
- `useAddSalesOrderPayment` / `useDeleteSalesOrderPayment` → invalidate `['mfg-sales-orders', docNo, ...]` but NOT the
  list root `['mfg-sales-orders']` (prefix matching cannot reach a *shorter* key) and not
  `['mfg-sales-order-detail', docNo]`. The SO list Balance column is `balance_centi_live`
  (pages/MfgSalesOrdersList.tsx:141-146, 232-238) → stale after recording a payment on the detail page.
- `useAddDeliveryOrderPayment` / `useDeleteDeliveryOrderPayment` → same shape: DO list root not invalidated.

### F10. UTC "today" defaults shift to yesterday before 08:00 MYT
`new Date().toISOString().slice(0,10)` is UTC; Malaysia is UTC+8. Pre-8am, every default date below is yesterday:
- Doc-date defaults on New forms: `DeliveryOrderNew.tsx:87`, `ConsignmentNoteNew.tsx:90`, `ConsignmentReturnNew.tsx:85`,
  `DeliveryReturnNew.tsx:84`, `GrnNew.tsx:170`, `PurchaseOrderNew.tsx:147`, `PurchaseInvoiceNew.tsx:139` (+ due date 260),
  `PurchaseReturnNew.tsx:152`, `SalesInvoiceNew.tsx:89`, `FlowDrawers.tsx:215,443`,
  `PurchaseConsignmentOrderNew.tsx:125`, `PurchaseConsignmentReceiveNew.tsx:127`, `PurchaseConsignmentReturnNew.tsx:125`.
- Payment date default: `components/PaymentsTable.tsx:165` (`newPaymentDraft.paidAt`).
- Date-range filters "today": `pages/AuditLog.tsx:118`, `pages/Outstanding.tsx:37-38`,
  `components/DetailListingShell.tsx:110-111`, `pages/SalesOrderDetailListing.tsx:575-576`.
- Day-grouping: `lib/audit-log-view.ts:6` (`ymd` of timestamps — entries 00:00-08:00 MYT group under the previous day).

The codebase already has both correct idioms: `new Date().toLocaleDateString('en-CA')`
(SalesOrderDetail.tsx:1630, SalesOrderNew.tsx:454) and `todayMY()` (+8h, SoFromProducts.tsx:36). Unify on one local-date
helper in `@2990s/shared`.

### F11. GrnNew create-then-post: post failure → re-Save creates a duplicate GRN
`pages/GrnNew.tsx:487-520` — `create.mutateAsync(...)` then `post.mutateAsync(createRes.id)`. If create succeeds and
post fails (network blip, 409 over-receipt), the catch shows "Save failed" and the form stays populated; clicking Save
again creates a **second** GRN draft and posts it → double receipt risk (plus an orphan unposted GRN). Stash
`createRes.id` after a successful create and retry post-only on the next click.

### F12. DataGrid: first drag-reorder / pin-left / right-click-hide reveals ALL default-hidden columns
`components/DataGrid.tsx:408-413` — `effectiveHidden` applies `defaultHidden` only while the layout is "pristine"
(`order.length===0 && hidden.length===0`). The Columns-popover path materializes defaults before toggling
(`toggleColumn`, 379-394), but these paths do not:
- `onDropHeader` (586-609) — any column drag-reorder sets `order` → pristine off → `hidden=[]` → the ~10
  default-hidden columns all appear at once;
- `pinLeft` (658-664) — same;
- `hideColumn` via header context menu (654-655) — hiding ONE column un-hides every default-hidden column.
Affects the 12 list pages using `defaultHidden` (MfgSalesOrdersList 22 uses, ConsignmentOrders 22, MfgDeliveryOrdersList 15, …).

### F13. SO maintenance/dropdown editor — `useUpdateAddon`/`useCreateAddon`/`useDeleteAddon` in `lib/mfg-products-queries.ts` show no `invalidateQueries` in their mutation bodies
(They map to `['addons']`/`['addons-all']` consumers.) Verify whether they rely on `setQueryData` — `pages/Addons.tsx`
has its own correct optimistic mutation (lines 49-83, with rollback), so these lib hooks may be the stale pair.
Flagged for verification rather than asserted (mutation bodies are long; only the invalidate-extraction pass showed none).

## P3 — hygiene / edge cases

### F14. Literal NUL byte (0x00) embedded in source
`pages/StockAdjustmentNew.tsx:169` — `` `${x.variant_key}\x00${x.batch_no ?? ''}` `` uses a **raw NUL character** as the
bucket-key delimiter (and presumably the matching `<option value>` builder). Runtime-functional, but the file is
detected as binary by grep/ripgrep/`file`/git tooling — it silently drops out of code search and diffs (this audit
only caught it because grep said "Binary file ... matches"). Replace with `' '` escape or a printable delimiter.

### F15. SalesOrderNew double-create window during post-create flush
`pages/SalesOrderNew.tsx:693-719` — after `create` succeeds, `onSuccess` awaits `flushPaymentDrafts` +
`flushPendingPhotos` while the user is still on the form; `create.isPending` is already false (button at 748 re-enables)
so a second click during the flush would POST a duplicate SO. Narrow window; add a local `finishing` flag.

### F16. `pages/Products.tsx:1362` — `window.prompt` default date uses UTC `toISOString` (same class as F10) for
compartment-rename `effectiveFrom`.
